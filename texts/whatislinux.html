Introdução

Ao longo do tempo muitas pessoas (geralmente acadêmicos) me pediram auxílio para instalar sistemas, encontrar alternativas de software, entender as alternativas de interface, manutenção básica… E essas pessoas tendem a ter um conhecimento base para usar as minhas documentações. Mas vez ou outra encontrei pessoas que queriam mesmo é entender o que é Linux e o porquê escolhi usar como meu sistema principal, qual é a diferença de utilizar Linux ao invés de Windows ou Mac. Eu sempre expliquei isoladamente, porém recentemente duas amigas me pediram auxílio não só para migrar mas também para entender a base do que é Linux, e ao notar que não tinha nenhuma documentação aqui a respeito disso resolvi fazer essa página.

Essa documentação parte do princípio que você não é nenhum acadêmico da área e também não vai iniciar de imediato a migração para o Linux, mas sim quer entender a base de como as coisas são quando o assunto é Linux.

Tenha uma coisa em mente: Aqui eu vou te ensinar a computação necessária para verdadeiramente entender o que o Linux é, muito disso não é algo especificamente DO Linux, então se você usa Windows ou Mac e acha tudo o que está aqui muito complexo provavelmente é devido a nunca ter aprendido computação na sua vida. E não tem nada de errado com isso.

Mas é importante ressaltar que existe uma diferença entre saber clicar em 2 ou 3 ícones ou mesmo escrever uma mensagem no Facebook e verdadeiramente entender o sistema que você está utilizando, é (em ordens de grandeza) muito mais fácil aprender o básico de se usar um sistema do que aprender o básico para entender o sistema (tanto quanto é mais fácil aprender a dirigir do que aprender mecânica de carros). Se algo aqui lhe parece muito distante provavelmente é porquê você nem mesmo entende essas condições no próprio Windows ou Mac, você só usa o básico.

O ponto aqui não é humilhar você, não me entenda mal por favor, é que você precisa entender a diferença entre aprender Linux e aprender computação. Cansei de ouvir pessoas falando "nossa, como Linux é difícil" quando eu estava explicando sobre coisas que são comuns da computação para todos os sistemas! Se a explicação é sobre compiladores, isso não é algo específico do Linux, sistemas usam compiladores! Se eu te explicar sobre sistemas de arquivo saiba que isso não é específico do Linux, sistemas operacionais tem sistemas de arquivo! Então se você começa com a mentalidade de "tudo isso é só do Linux? Que coisa difícil" já era, mas se você se focar em aprender computação e através dela entender Linux os seus resultados serão completamente diferentes.

Primeiramente, o que é Linux?

Creio que a definição mais clara do que é Linux é que Linux é uma tecnologia (ah vá, é mesmo?), livre e de código aberto (FLOSS), e também modular. Linux pode ser uma parte do sistema do seu smartphone (caso você use Android) assim como pode ser um sistema completo para o servidor da sua empresa (caso utilizem Ubuntu). Algumas pessoas tratam o Linux como UM sistema operacional, algumas pessoas tratam o Linux como UM kernel. E se alguém te disse isso antes, saiba que essa pessoa está errada!

Para entender o erro dessa afirmação precisamos entender o que é um sistema operacional e o que é o kernel. E acredite: É bem mais fácil entender isso do que parece.

O que é um sistema operacional?

Podemos entender sistemas como o conjunto de elementos que interagem entre si de maneira lógica para atingir um objetivo, parece complicado? Imagine como funciona um relógio: Você tem diversas peças, todas elas interligadas com o fim de te mostrar as horas (eu disse que era mais fácil do que parece). Na computação, assim como dividimos o que é software (a parte lógica / os programas, documentações e licenças) e o que é hardware (a parte física / as peças), e dividimos cada componente de hardware (placa mãe, hdd, placa de vídeo, fonte…) também dividimos cada componente de software (ou seja, cada tipo de programa).

Ainda que você seja um completo leigo em computação você já deve saber que computadores funcionam através de código binário, mas você certamente não utiliza seu computador com um interruptor alterando cada pulso elétrico, você usa teclado e mouse e lê no seu monitor.

Na computação não tem magia, se VOCÊ usuário não precisa modular cada pulso e comunicar cada fase de um processo de inicialização e cada alteração no disco pro seu computador subir um sistema é porque alguém faz isso por você, e o cara responsável por fazer isso por você se chama Sistema Operacional.

Se sistema é o conjunto de elementos que interagem entre si de maneira lógica para atingir determinado objetivo, Sistema Operacional é o software que tem por objetivo ser a ponte entre o que o usuário faz e o que a máquina entende. Assim como o harware também é composto de outras peças (placa de vídeo tem seu próprio chip de memória, seu próprio processador) o sistema operacional também tem as suas peças, mas já que você não veio aqui pra fazer um artigo de doutorado você só precisa entender superficialmente 2 componentes: Espaço de usuário e Kernel.

Espaço de usuário é a área do sistema operacional voltada para atender o usuário (duh), como por exemplo o gerenciamento de janelas e os elementos gráficos.

Kernel é a área do sistema operacional voltada para atender o próprio computador, como gerenciamento de memória, sistema de rede, interfaces virtuais para comunicação com dispositivos.

E aqui vai o primeiro alerta dessa documentação: Mais de 80% do sistema operacional é invisível aos olhos do usuário final, então muito do que você julga ser o sistema operacional provavelmente não é ou tende a não ser uma parte relevante/intrínseca do sistema.

E qual a razão dessa confusão toda? A razão é que quando você fala “Linux” você designa uma tecnologia de kernel completa, monolítica, modular, multitarefa, multiusuário, multi arquitetura , e baseado em Unix (já explico o que é Unix), que é usada em diversos dispositivos e sistemas.

Então sempre que alguém por aí quer fazer algum dispositivo que precise de um kernel, a tendência é simplesmente utilizar o Linux mesmo, funciona em basicamente tudo.

Seu Android não é “O LINUX”, mas ele é Linux também, seu roteador não é “O LINUX”, mas ele provavelmente tem partes de Linux também.

Então, diferente do Mac ou Windows que você tem que instalar por inteiro e você só instala em arquiteturas de pc (x86/x64) ou de smartphone (ARM), o Linux você pode usar modularmente onde quiser.

O Linux também tem outros componentes que vão além do kernel, como init system, bootloader, bibliotecas… Porém, pela própria natureza da tecnologia ela não é UM sistema operacional completo, fechado.

E isso não é um problema (como muitos usuários de BSD tentam insistir) FAZ PARTE DO DESIGN DO LINUX ser assim! Pegue como precisa, implemente como quiser e onde quiser.

Então daí a confusão, tem quem não sabe que existem mais componentes e por isso acha que Linux é apenas um kernel, e tem quem usa alguma implementação de sistema baseado em Linux e acha que Linux é um sistema. Porém ele não é nem um nem outro, lembre-se:

Linux é uma tecnologia modular e livre. Não é um único sistema completo, e nem somente um kernel.

Livre? Código aberto?

Quando você utiliza o Windows (legalmente), ou é porque você comprou um computador que já veio com o Windows instalado ou porque montou o seu computador e depois comprou uma licença de uso do Windows por 1500 pila e baixou o sistema do site da Microsoft e instalou por conta própria no seu computador.

Quando você utiliza o MacOS (legalmente) é porque você comprou um computador da Apple que já veio com o sistema. Esses sistemas tem dono! Você não pode modificar o que quiser e muito menos redistribuir ou revender!

Isso se dá por uma parte importantíssima dos programas de computador que a maioria das pessoas sequer sabe que existe: a licença de uso.

Todo programa que você usa no seu computador, seja sistema ou seja aplicativo tem uma licença de uso que determina o que você pode e o que não pode fazer, e mais importante ainda determina o que o programa pode ou não fazer com você e o seu computador, sabe aqueles termos de uso que você ignora? Então…

Cada programa tem sua licença e caso você queira se formar em computação você provavelmente vai ter algumas aulas completamente focada em licenças de software. Mas como você veio aqui para aprender o básico, então vamos nos focar no que você precisa aprender sobre licenças:

Uma licença pode ser…

Proprietária, o programa tem dono e não pode ser alterado e/ou redistribuído.

OU

Livre, em resumo o software é considerado “software livre” se ele tem 4 liberdades básicas, que são

  • A liberdade de usar o programa como desejar;
  • A liberdade de estudar e alterar o programa (para isso o código fonte do programa deve ser aberto);
  • A liberdade de redistribuir o programa;
  • A liberdade de distribuir publicamente as alterações e modificações criadas por você.

Existe toda uma discussão entre softwares proprietários e softwares livres, e as liberdades do software livre vão bem além disso, mas grosso modo essa é a diferença.

Com relação ao código, você sabe que o computador executa código binário, então para se tornar uma aplicação funcional o código escrito por um humano em alguma linguagem de programação precisa passar por uma ferramenta chamada compilador para transformar esse código “humano” em código de máquina. Entenda o código do programa como a “receita” para se fazer o programa.

Basicamente faz isso:

função (calcular) {
valor a = 5;
valor b = 2;
valor c = a + b;
retornar c;
}
Virar isso: 01100110 01110101 01101110 11000011 10100111 11000011 10100011 01101111 00100000 00101000 01100011 01100001 01101100 01100011 01110101…

(sim, é pseudocódigo com conversão de texto pra decimal e foi só pra dar exemplo…)

Sendo assim o software pode ter…

O código fechado, e assim o programa é distribuído apenas em formato de executável, sem receita de bolo, de maneira que você não pode ver o que o programa faz no seu computador você pode apenas executá-lo.

OU

O código aberto, você ver toda a “receita” do programa e entender tudo que o programa é capaz de fazer e como ele interage com o seu computador.

Então entenda, o Windows é um produto, assim como o Macbook, você compra e consome. O Linux é como um livro de domínio público, você pode ter acesso àquele conteúdo e interagir com ele como quiser.

Essa é a razão que faz com que chamar o Linux de sistema GRATUITO esteja completamente errado! Windows e Macbook são produtos, são compráveis, talvez alguém te dê um gratuitamente mas eles não deixam de ser propriedade das empresas que os fizeram.

O Linux em si não é um produto, ele não é vendido, ele é livre!

No mundo da computação, gratuito é uma designação do software proprietário de custo 0! Essa não é a condição do Linux!

Lembre-se: Linux não é um sistema “de grátis”, ele simplesmente não é um produto, ponto.

Comparar os incentivos do Windows como software pago com o Linux como software livre é como comparar os incentivos no pagamento de uma refeição em um restaurante famoso com os incentivos para comer a comida da sua avó. Você não come a comida da sua avó “por ser de grátis”, você não come a comida da sua avó por uma questão financeira, você come por outros valores.

O que são distribuições?

Como eu disse anteriormente, o Linux em si não é um sistema operacional completo, mas não quer dizer que não existam sistemas Linux completos para uso. Caso você queira utilizar Linux como seu sistema operacional, você não precisa baixar código, compilar e “completar” o sistema (a menos que você deseje), você pode simplesmente baixar uma mídia de instalação de alguém que redistribui o Linux como sistema, e essas distribuições nós chamamos de distros (bem intuitivo né?).

Existem diversas empresas e comunidades que usam o Linux, o modificam e redistribuem as suas instalações para diversos propósitos, seja para criar um servidor de mídia (como o LibreELEC) ou fazer do seu computador uma central de jogos (como o Batocera).

Se eu quiser iniciar a entender todas essas? O que eu preciso saber inicialmente?

Que o Slackware é considerada a mais Unix das distribuições. Tudo aqui você faz na mão, excelente pra estudos mas não recomendaria para iniciantes.

Que o Debian é focado em estabilidade e é muito usado em servidores, não recomendo para usuários finais mas é “pai” do Ubuntu que eu até interessante para usuários finais.

Que o Red Hat é uma distribuição paga (SIM, PAGA!) basicamente a distribuição com maior produção e adoção de novas tecnologias, não recomendo para usuários finais mas é pai do Fedora que é até interessante para usuários finais.

Que o Arch Linux é uma distribuição rolling release, ou seja, as atualizações são implementadas assim que são disponibilizadas. Não recomendo para usuários iniciantes embora seja excelente para usuários avançados, pai do conhecido Manjaro que é uma excelente opção para usuários médios.

Que o Gentoo é uma distribuição focada em compilação local dos aplicativos (ou seja, você baixa o código e compila os programas para depois usar), não recomendo nem para usuários médios mas para usuários muito avançados já que permite uma customização sem igual.

Sendo assim as distribuições podem ser:

  • Voltadas para usuário final (Fedora) OU uso específico (Red Hat);
  • Versionadas (Debian) OU rolling release (Arch);
  • Baseadas em binários (Debian) OU em compilação (Gentoo);
  • Focadas em estabilidade (Debian) OU em ser ponta de lança (Arch);
  • Automatizadas (Debian) OU “customize e faça você mesmo” (Slackware).

Se você considera qualquer distribuição nesses termos e sabe em que ela foi baseada você entende muito em como se virar nela, da mesma forma que se você entende de Android você entende de Lineage, se você entende de Debian você entende de Ubuntu.

Eu já sei o que posso instalar, mas como eu instalo?

Diferente do Windows que é baseado em você ir na página do desenvolvedor, baixar um .exe, dar 2 cliques, permitir acesso administrativo, clicar em next next next e rezar pra não ter vírus, no Linux a grande maioria dos sistemas para usuário final já tem uma lojinha de aplicativos para você baixar o que precisa.

Então caso queira instalar alguma coisa pra fazer algo no seu computador, para de sair fazendo pesquisa aleatória no Google ou Youtube, visite o alternativeto, veja as opções de software e busque esse mesmo software na lojinha de aplicativos do seu sistema.

E se não tiver na lojinha? O que são pacotes?

Quando você quer instalar um aplicativo no Windows você utiliza um arquivo .exe ou .msi, esses são pacotes de instalação.

Cada distribuição Linux tende a ter o seu próprio gerenciador de pacotes e o seu tipo de pacote, assim como você tem versões de aplicativos específicas para Windows XP e Windows 7, no Linux os pacotes .deb são para a base Debiam assim como os .rpm são para a base Red Hat. Embora esses pacotes tendem a funcionar em distribuições que se baseiam nos padrões dos pacotes (.deb funcionar em Ubuntu já que Ubuntu é baseado em Debian) eu recomendo você utilizar pacotes universais.

Algumas vezes a sua distribuição pode não empacotar o software que você precisa devido a falta de um mantenedor ou coisa do tipo (e obviamente não estará disponível na lojinha), nesse caso você ainda tem a opção de tentar instalar o software através de um .deb (para distribuições baseadas em Debian) ou .rpm para distribuições baseadas em Red Hat, mas eu recomendo que você utilize Appimage, Flatpak ou Snap (mas especialmente Flatpak).

Esses pacotes são como os arquivos .dmg no Mac, porém no caso do Linux eles funcionam independente da distribuição Linux que você usa, ou seja, se a distribuição que você utiliza não empacota a última versão do Retroarch basta você instalar a versão Flatpak com o comando flatpak install flathub org.libretro.RetroArch que vai rodar.

Você citou Unix, o que é isso?

É difícil explicar a importância do Unix em menos de 10 páginas, mas em resumo o Unix é uma família de sistemas operacionais, multiuso e multitarefa criado no Bell Labs por pessoas como Ken Thompson, e Dennis Ritchie.

Diferente do Windows que foi desenvolvido já focado no usuário final em âmbito comercial, o desenvolvimento do Unix se confunde muito até mesmo com o desenvolvimento da Linguagem C (linguagem de referência para programação de sistemas ATÉ HOJE), o Unix possibilita uma portabilidade e modularidade muito alta e teve diversas variantes tanto no âmbito comercial como no âmbito acadêmico (sem contar a quantidade maior ainda de sistemas que utilizaram o Unix como base).

Os sistemas Unix são caracterizados por um design brutalmente simples e modular. E ao invés de simplesmente dar lucro, o Unix tem toda uma filosofia de uso e desenvolvimento por trás. Existem diversos livros sobre essa filosofia, mas eu vou tentar explicar da seguinte forma, em termos de desenvolvimento:

  • Desenvolva programas que façam uma coisa só e que façam muito bem, faça um novo programa para uma nova função ao invés de complicar velhos programas com funções novas.
  • Desenvolva os programas de modo que o resultado da execução de um programa se transforme na entrada de dados para outro programa.
  • Faça programas que lidem com fluxo de textos, pois essa é a interface universal.

Em termos de utilização, ao invés de ter um sistema massivo e monolítico o Unix se focava num modelo mais baseado em Kernel, Shell e Aplicação.

O Kernel se comunicava com os dispositivos, o Shell era a interface de comunicação com o Kernel e as Aplicações se comunicam com o Shell. Qual a vantagem disso?

Imagine que você tem uma lista de convidados para conferir, você precisa de todos os nomes dessa lista sejam únicos, que estejam em letras maiúsculas, em ordem alfabética, que sejam numerados e que por fim sejam comparados com uma primeira lista. Seria extremamente complexo de se fazer um programa pra fazer tudo isso, porém…

  • É fácil criar um programa de comparação de linhas;
  • É fácil criar um programa pra deixar tudo maiusculo;
  • É fácil criar um programa pra deixar uma lista em ordem alfabética;
  • É fácil criar um programa para numerar linhas;
  • E também é fácil criar um programa que mostre a diferença entre 2 arquivos…

Se você tem uma arquitetura próxima desse módulo dito acima (kernel, shell, aplicação) basta mandar esse comando no seu terminal:

compara_linhas Lista1.txt > maiúsculo > ordem_alfabética > numerar > diferente_de Lista2.txt

E Voilà! (sim, é um pseudocódigo também...)

E antes antes disso? Você precisava de um editor de texto específico para abrir o texto de um formato específico, esse editor teria que ter as próprias funções de comparação, ordenação e enumeração, essas funções só seriam funcionais nesse programa, elas teriam que ser chamadas com as próprias instruções de manipulação do sistema de arquivos, se você queria o resultado desse texto como saída para algum dispositivo (como por exemplo uma impressora) você teria que estender esse programa para possibilitar que ele se comunique com o dispositivo… Agora imagina ter um sistema TODO com esse retrabalho…

Entende como a arquitetura do Unix é muito mais elegante? Você não estende o seu programa para fazer tudo, você faz com que seu programa use o Shell para enviar informações para outro programa que se vire com o dispositivo em questão. É muito mais eficiente.

Essa “filosofia”, essa maneira de programar e usar o computador influenciou fortemente tanto o Linux quanto os BSDs (especialmente os BSDs).

Você não gostou da interface gráfica do seu Linux? Troca ela, o sistema não depende dela.

Você não gostou do sistema de arquivos do seu Windows? Que pena, o sistema é “casado” com o sistema de arquivos.
Você não gostou do sistema de arquivos do seu Linux? Troca ele, o sistema não depende dele.

Você não gostou do applet de rede do seu Windows? Que pena, o sistema é “casado” com o applet de rede.
Você não gostou do applet de rede do seu Linux? Troca ele, o sistema não depende dele.
